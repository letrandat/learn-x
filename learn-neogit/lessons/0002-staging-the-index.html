002: Staging — The Index Demystified

Goal: Master staging/unstaging in neogit while understanding git add and git restore --staged.

The Three Areas of Git

Git has three areas you must understand:

  1. Working tree — your edited files on disk
  2. Index (staging area) — changes queued for the next commit
  3. Repository — committed snapshots
Key insight: Only changes in the index get committed. Changes in the working tree alone are not saved until you stage and commit.

neogit staging workflow

The status buffer shows two critical sections:

  • Unstaged — changes in working tree, not yet in index
  • Staged — changes ready to be committed

Stage a single file

Move cursor to an unstaged file, press s:

:Neogit → cursor on file → s

CLI equivalent: git add <file>

Unstage a single file

Move cursor to a staged file, press u:

:Neogit → cursor on file → u

CLI equivalent: git restore --staged <file>

Stage all unstaged

Press S (capital S) from status buffer:

:Neogit → press S

CLI equivalent: git add .

Stage all (including new files)

Press C-s (Ctrl+s) from status buffer:

:Neogit → press C-s

CLI equivalent: git add -A

Exercise: Stage, modify, unstage

  1. Open neogit: :Neogit
  2. Stage a file with s — press $ to see git add
  3. Modify the same file in another buffer
  4. Back in neogit, you'll see it in "Unstaged" — press s again to re-stage
  5. Try u to unstage — see git restore --staged in history
  6. Compare: git diff vs git diff --cached

Quick quiz

Q: You staged app.js but want to unstage it. Which key?

Reveal Answer
Answer: u — the counterpart to s that unstages the item under the cursor.
