Lesson 0003

Worked example: messy notes → PPTX

One real deck, start to finish. Every artifact is a file in examples/ you can open and run.

We'll build an 8-slide internal pitch — "Put Redis in front of our product API" — the exact shape of deck you'd ask an LLM to make from notes + code. Follow the five files in examples/ as you read.

1

The raw input

examples/01-raw-notes.md

What you'd paste into the LLM: unstructured notes, a code snippet, plus the meta-facts that decide everything — audience, time slot, and the goal ("backend team, 10 min, get buy-in for a spike"). Always include those; they're your spec.

2

Lock the outline — before any Marp

examples/02-outline.md

Prompt used:

Draft a slide-by-slide OUTLINE only — no Markdown. One line each:
title + one-sentence purpose. ~8 slides, arc hook→problem→solution→proof→ask.

You get plain text you can scan in 10 seconds and fix cheaply:

1. Title + the ask
2. The pain — p99 ~800ms, DB hammered
3. The insight — 80/20 repeat reads
4. The plan — cache-aside
5. The code — get_product snippet
6. The risks
7. The mitigations
8. Payoff + ask — approve a spike

This is the step that saves you. If the story is wrong, you fix one line here — not 8 broken slides later. Approve it explicitly before moving on.

3

Generate the Marp Markdown

examples/deck.md

Now — and only now — you ask for syntax, handing over the rules block. The front-matter sets the whole deck:

---
marp: true
theme: gaia
size: 16:9
paginate: true
---

Title slide uses spot directives (note the underscores) so they touch only slide 1:

<!-- _class: lead -->
<!-- _paginate: skip -->

# Putting Redis in Front of Our Product API

Open examples/deck.md to read all 8 slides — code fence, blockquote, and a per-slide _footer included.

4

Catch what the weak model got wrong

A low-reasoning model's first draft of the title slide typically looks like this — and it's subtly broken in two ways:

---
marp: true
theme: corporate          ← invented theme; silently ignored
---

<!-- class: lead -->       ← no underscore: leaks to EVERY slide

# Putting Redis in Front of Our Product API

Your review catches both from the cheat-sheet checklist. The fix:

---
marp: true
theme: gaia               ← real theme
---

<!-- _class: lead -->      ← underscore: this slide only

# Putting Redis in Front of Our Product API

This is the whole skill. You didn't need to write Marp — you needed to spot the invalid theme and the missing underscore, then tell the model exactly what to fix. That's your job as editor-in-chief.

5

Preview, then export

See it before you trust it. Fastest paths:

# live preview window, auto-reload on save
npx @marp-team/marp-cli@latest -p -w examples/deck.md

# or paste the file into https://web.marp.app/

Happy? Export to PowerPoint:

# PPTX as images — pixel-perfect, keeps notes, NOT editable text
npx @marp-team/marp-cli@latest examples/deck.md --pptx

Your setup: you have Node ✓ but not LibreOffice, so the editable path (--pptx-editable) is off the table for now — default --pptx (images) is your route. PPTX/PDF/PNG export also needs Chrome, Edge, or Firefox installed (Marp renders through it).

Heads-up on your local setup: your package.json has only @marp-team/marpit (the engine — no themes, no export). The npx … marp-cli commands above sidestep that. If you want a local install: npm i -D @marp-team/marp-cli.

6

The loop, if a slide is off

Don't regenerate the whole deck. Point at one slide, name the bug:

Slide 5's code overflows the slide. Shorten the comments and drop blank
lines so it fits; return the full file.

Re-preview. Repeat until clean. That's the entire lifecycle.

Want to actually run it? Open examples/deck.md, then run the preview command above. If anything errors (browser not found, etc.), tell me the message and I'll get you unstuck. I can also generate a second deck from your real notes so you practice the review step on fresh output.
