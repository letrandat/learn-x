Lesson 0002

From Markdown to PPTX: the real lifecycle

Your instinct was close. Here's the corrected, full pipeline — and how to drive the LLM through each stage.

Your mental model, graded

"Gather context with the LLM → generate a Marp .md → preview in browser → export PPTX."

~80% right. Two missing pieces that matter most because your model is weak:
  • A locked outline before any Markdown. Never let a low-reasoning model do "decide the story" and "write Marp syntax" in one shot — it'll botch both.
  • PPTX is rasterized images by default — each slide is a picture, not editable text. This changes how you plan the whole thing.

The full lifecycle

  1. You — before touching the LLM Set the spec Audience, length (~N slides), goal, theme (default/gaia/uncover), aspect (16:9), and editable-or-image PPTX? (see §PPTX). Write these down — they become constraints you hand the model.
  2. You + LLM — conversation Gather context Feed your .md notes + code. Ask the LLM to summarize and ask clarifying questions. Don't ask for slides yet.
  3. LLM — plain text, no Marp yet Agree the outline / storyboard One line per slide: title + 1-sentence purpose. You review and lock this. Cheap to fix here, expensive later. This is the step you were missing.
  4. LLM — now syntax Generate Marp Markdown Give it the rules (front-matter with marp: true, one slide per ---, valid theme, scope underscores). Use the prompt template → reference 0002.
  5. You — eyeball Preview & verify Paste into web.marp.app, or marp -p -w deck.md, or VS Code preview. Run the failure-mode checklist. Catch overflow, leaked styles, broken images.
  6. You + LLM — iterate Fix specific slides Point at the broken slide, name the bug ("style leaking — add the _"). Tight loop. Re-preview.
  7. You — CLI Export PPTX marp deck.md --pptx (needs Chrome/Edge/Firefox installed). See §PPTX for the editable trap.

The hinge is steps 3 vs 4: outline (thinking) is separated from Marp syntax (formatting). A weak model that conflates them produces a confident, broken deck.

How to prompt at each stage

Don't write one giant prompt. Drive it in turns — each with a narrow job:

Stage 2  "Here are my notes and code. Summarize the key points.
          Ask me up to 5 questions before we plan slides."

Stage 3  "Draft a slide-by-slide OUTLINE only. One line each:
          slide title + one-sentence purpose. ~12 slides.
          No Markdown yet."

Stage 4  "Now convert THIS outline to Marp Markdown.
          Rules: start with front-matter (marp: true, theme: gaia,
          paginate: true). One slide per ---. Use _class: lead only
          on the title slide. No invented directives. Keep each slide
          under ~6 bullets so nothing overflows."

Stage 6  "Slide 7's dark background leaks onto 8-10. Fix by making it
          a spot directive (_backgroundColor)."

The full copy-paste rules block lives in reference 0002 — paste it once and the model has the guardrails.

The PPTX trap (read this twice)

marp deck.md --pptx does not give you editable PowerPoint text. Each slide is exported as a rendered background image — you can't click into a bullet and edit it. [marp-team discussion #82]

CommandYou getCost
--pptxSlides as images. Pixel-perfect to your Marp design. Presenter notes preserved.Not editable in PowerPoint
--pptx --pptx-editableEditable text/shapesExperimental. Needs LibreOffice installed. Lower visual fidelity; complex CSS may break; no presenter notes
Decision rule (set in step 1): if the deck just needs to present and look exactly like your Marp design → default --pptx. If a colleague must edit text in PowerPoint afterward → --pptx-editable + accept the fidelity hit, or reconsider whether Marp is the right tool for that handoff.

Both need a browser (Chrome/Edge/Firefox) installed — Marp uses it to render. Same for PDF and PNG export.

One-line recap

Spec → context → locked outline → Marp md → preview/verify → iterate → --pptx (images, unless you opt into editable).
