Lesson 0001

The Marp Mental Model

Just enough to read an LLM's slide deck and know if it's any good.

Marp = Markdown → slides. You write a normal-ish Markdown file; Marp turns each chunk into a slide. Your job isn't to author it — it's to review what your LLM produces. To do that you need three things: the layer model, the anatomy of a deck, and a nose for the handful of mistakes a weak model always makes.

1 · Three layers (so you know what to blame)

  • Marpit — the engine. Defines the core syntax: slide breaks, directives, image/background syntax. marpit.marp.app
  • Marp Core — adds the nice stuff: built-in themes (default, gaia, uncover), math, emoji, auto-fit headings.
  • Marp CLI / VS Code — the tools that turn your .md into HTML / PDF / PPTX, or preview it live.

When something's wrong, this tells you which doc to paste at the LLM: syntax/directive bug → Marpit docs; theme/math bug → Marp Core; export bug → Marp CLI.

2 · Anatomy of a deck

---
marp: true          ← REQUIRED. Turns the file into a deck.
theme: gaia
paginate: true
---

# Slide one

Some words.

---                 ← this ruler = new slide

<!-- _class: lead -->   ← directive, THIS slide only (note the _)

## Slide two

Two things do all the structural work:

  • Front-matter (the top --- ... --- block) — deck-wide settings. Must contain marp: true.
  • The --- ruler — every one after the front-matter starts a new slide.

3 · Directives have a scope — this is what LLMs break

A directive is a setting written as an HTML comment (or in front-matter). The scope depends on how it's written:

<!-- theme: gaia -->        global — whole deck
<!-- paginate: true -->     local  — this slide AND every slide after
<!-- _paginate: false -->   spot   — this ONE slide only  (leading _)

The leading _ underscore means "this slide only." Forget it, and a one-slide style leaks across the whole deck. This is the most common Marp bug a low-reasoning model makes. When you see a style "stuck on" later slides — hunt for a missing underscore.
