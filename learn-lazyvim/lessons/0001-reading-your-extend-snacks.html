Reading your extend-snacks.lua

One file, fully understood — what each line changes, and the merge model that makes "override only what you want" work.

You already edit this file. The goal of this lesson is narrow: by the end you can point at any line and say what it changes versus the default, and why. Here is your current file (repos/dotfiles/nvim/.config/nvim/lua/plugins/extend-snacks.lua):

return {
  "folke/snacks.nvim",        -- ① which plugin this spec extends
  opts = {                       -- ② your overrides, deep-merged onto defaults
    terminal = {                   -- ③ the terminal MODULE
      win = {
        position = "float",
        width  = 0.9,
        height = 0.9,
        border = "rounded",
      },
    },
    picker = {                     -- ④ the picker MODULE
      layouts = {                  -- ⑤ how the window is sized
        default = {
          layout = { width = 0.9, height = 0.9 },
        },
      },
      sources = {                  -- ⑥ per-source behaviour
        files = {
          follow  = true,
          exclude = { "node_modules", ".git" },
        },
        grep = {
          follow  = true,
          exclude = { "node_modules", ".git" },
        },
      },
    },
  },
}

① + ② The merge model — the one idea to keep

This file is a LazyVim plugin spec. Because "folke/snacks.nvim" is already defined by LazyVim, your spec doesn't redefine it — it extends it. The opts table is deep-merged, so your effective config is three layers stacked:

snacks defaults ┐ ▼ overridden by │ deep merge, key by key LazyVim defaults │ (you only write the keys you ▼ overridden by │ want to change) YOUR opts ────────────►┘ = effective config

That's why the file is tiny: you are not configuring snacks, you are diffing against it. Every key below is a deliberate deviation from a default. Keep asking: "what was the default before this line?" — that question is the whole skill. (See glossary: opts / module / source.)

③ The terminal module — float instead of split

terminal is one snacks module. Its default with no command is a bottom split. Your win.position = "float" + 0.9 / 0.9 + "rounded" turns the toggle terminal into a big centered floating box. The 0.9 values are proportions of the screen — numbers between 0 and 1 mean "fraction of the screen", integers would mean "this many characters". That single sizing rule shows up again two blocks down.

④ + ⑤ The picker module & its layout

picker is the module behind <leader>ff, <leader>/, and most of LazyVim's search keys. Snacks normally picks a layout preset by terminal width (a wide default with preview, or a stacked vertical on narrow terminals). You aren't replacing that logic — you're only resizing the default preset to 0.9 / 0.9 so the picker fills most of the screen. Same proportion rule as the terminal.

⑥ Sources — where picker behaviour lives

A source is a named picker type: files lists filenames, grep does live full-text search. Each has options. Your file touches two:

optiondefaultyour valueeffect
followfalsetruefinder follows symlinks into their targets
exclude{}node_modules, .gitfinder always drops these paths

Note what you did not set: hidden and ignored both stay at their default false. ignored = false is the line that normally hides node_modules for you — because it tells the finder to skip anything in .gitignore.

The gotcha you just hit ignored = false relies on the backend (fd) reading .gitignore — and fd only reads .gitignore inside a git repository. Your ~/.config/opencode/ folder isn't a git repo, so .gitignore was ignored and node_modules leaked into <leader>ff. Adding exclude fixes it unconditionally: it passes --exclude node_modules straight to fd/rg, no git required. That's exactly the two lines you added.

Your config sets neither hidden nor ignored on the files source. In a normal git repo, what shows up in <leader>ff?

Both default to false: hidden=false drops dotfiles, ignored=false drops gitignored paths. Your exclude is just an extra, git-independent filter on top.

Try it — make the merge model concrete (2 min)

Inside Neovim, prove to yourself these are deviations from a default:

  • Run :lua Snacks.picker.files({ hidden = true }) — watch dotfiles appear, then vanish again on a normal <leader>ff. You just toggled one source option live.
  • Run <leader>ff in a real git repo vs. in ~/.config/opencode/ — same config, different result, because of the git-repo caveat above.
  • Open :Snacks picker (or <leader>sk for keymaps) to see how many sources exist beyond files/grep.
