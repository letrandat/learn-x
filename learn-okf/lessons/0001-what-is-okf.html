Lesson 0001 • Open Knowledge Format

The Context Assembly Problem and OKF

Why AI agents need a shared language for metadata, and how OKF builds it.

Introduction

Modern large language models (LLMs) are incredibly capable. They can write code, build plans, and analyze datasets. However, they are fundamentally limited by context. Without access to the correct, up-to-date internal details of an organization, foundation models cannot produce accurate and actionable results.

The Context Assembly Problem

In most companies, metadata is fragmented. It lives across database schemas, wiki pages, docstrings, and slack channels. Every catalog vendor has its own SDK or API. This means every agent developer must solve the same "context assembly" problem from scratch, and knowledge remains siloed.

What is the Open Knowledge Format?

The Open Knowledge Format (OKF) is an open-source, vendor-neutral specification (currently v0.1) that formalizes the "LLM-Wiki" pattern into a portable and interoperable structure.

OKF relies on three core simplicity principles:

  • Just Markdown: Concepts are represented as human-readable markdown files, editables in any tool.
  • Just Files: Shipped as standard directories, git repositories, or tarballs.
  • Just YAML Frontmatter: Minimal metadata fields used to describe the concepts.

Concept Documents and Graph Relations

An OKF Bundle is a folder containing concept documents. Each file represents a concept (e.g. a table, a metric, a runbook, or an API).

A concept document is a markdown file with YAML frontmatter at the top: 

---
type: table
title: core.users
description: Authoritative register of all user accounts.
resource: bigquery://my-project.core.users
tags: [pii, core]
timestamp: 2026-07-06T21:45:00Z
---
# core.users
This table lists all registered accounts...

Concepts link to each other using standard markdown links (e.g., [users](table/users.md)), turning the directory structure into a relational graph.

Key Principle: Minimally Opinionated

The spec requires exactly one field in the YAML frontmatter: type. This ensures the standard governs interoperability, not the exact schemas you choose to model.
