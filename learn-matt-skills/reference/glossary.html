Reference · Glossary

Skills system glossary

The shared vocabulary for this course. Once a term is defined here, every lesson uses it the same way. Print this; keep it beside you.

Skill

A folder holding a SKILL.md (YAML frontmatter + instruction body) and optional bundled files. Loaded into the agent's context to steer one task. Not code; not a framework.

User-invoked skill

Runs only when you type /name. Job: orchestrate a flow. Marked with disable-model-invocation: true. E.g. /grill-me, /triage, /setup-matt-pocock-skills.

Model-invoked skill

Can be triggered by you or reached for automatically by the agent when the task fits. Job: hold reusable discipline. E.g. tdd, diagnosing-bugs, domain-modeling.

Composition rule

A user-invoked skill may invoke model-invoked skills, but never another user-invoked one. Keeps orchestration flat — one conductor per flow.

Progressive disclosure

The agent sees only each skill's description until one matches; only then is the full body (and its bundled files) loaded. Keeps context cheap. Why the description must say what + when.

Bundled resource / seed template

An extra file in a skill folder, referenced by relative link in the body and read only when needed. Setup's five .md templates are seed content copied into your repo.

Frontmatter

The YAML block at the top of SKILL.md: name (→ slash command), description (relevance signal), optional disable-model-invocation, argument-hint.

Issue tracker config

docs/agents/issue-tracker.md — where issues live (GitHub / GitLab / local markdown / other) and the concrete commands to use. Read by /triage, /to-issues, /to-prd.

PRs as a request surface

A yes/no flag in the issue-tracker config. If yes, /triage pulls external pull requests into the same queue and labels as issues.

Triage labels

docs/agents/triage-labels.md — a mapping from five canonical roles to your repo's real label strings, so /triage applies existing labels instead of creating duplicates.

ready-for-agent

The pipeline's endgame: an issue specified completely enough that an away-from-keyboard autonomous agent can pick it up with zero additional human context.

Domain docs config

docs/agents/domain.md — tells skills to read CONTEXT.md + docs/adr/ before exploring, and whether the repo is single- or multi-context.

CONTEXT.md

The project's glossary / ubiquitous language. Lets the agent name concepts concisely and consistently. Created lazily by /domain-modeling, not upfront.

ADR (Architectural Decision Record)

A dated note under docs/adr/ capturing one non-obvious decision and its reasoning. Skills must flag, not silently override, conflicts with an existing ADR.

Single- vs multi-context

Single: one CONTEXT.md at root (most repos). Multi: CONTEXT-MAP.md at root pointing to per-context CONTEXT.md files (monorepos).

PRD (to-prd)

Product Requirements Doc. /to-prd (user-invoked) synthesises the current conversation into a PRD — no interview — and publishes it to the tracker with the ready-for-agent label. Records decisions (problem, solution, user stories, implementation/testing decisions, scope); bans file paths and code snippets.

Seam (Feathers)

A place where you can alter behaviour without editing in that place — where a module's interface lives. For testing: prefer existing seams, use the highest possible, fewer is better (ideal: one). Chosen in to-prd step 2; tdd tests at these seams. Avoid: "boundary" (overloaded with DDD).

Module / Interface / Implementation

Module — anything with an interface + implementation (fn, class, package, slice); scale-agnostic. Interface — everything a caller must know (types, invariants, ordering, error modes, perf), not just the signature. Implementation — what's inside. Avoid: component, service, API.

Depth (deep vs shallow)

Leverage at the interface: behaviour exercised per unit of interface learned. Deep = lots of behaviour behind a small interface (goal). Shallow = interface nearly as complex as implementation (avoid). Not the impl-lines÷interface-lines ratio — that rewards padding.

Leverage / Locality

Leverage — what callers gain from depth: more capability per unit of interface learned. Locality — what maintainers gain: change, bugs, knowledge concentrate in one place. Fix once, fixed everywhere.

Adapter

A concrete thing satisfying an interface at a seam — describes role (what slot it fills), not substance. Rule: one adapter = a hypothetical seam; two = a real one.

The deletion test

Imagine deleting a module. If complexity vanishes → it was a shallow pass-through. If complexity reappears across N callers → it was earning its keep (deep). "Concentrates complexity" is the good signal.

improve-codebase-architecture

The maintain-stage user-invoked orchestrator. Scans for deepening opportunities, presents a visual HTML report (Tailwind+Mermaid, before/after, strength badges) to the temp dir, then runs /grilling + /domain-modeling on the candidate you pick. Built on codebase-design vocabulary; informed by CONTEXT.md + ADRs.

Vertical slice / tracer bullet (to-issues)

An issue that cuts a thin, complete path through every layer (schema→API→UI→tests), demoable on its own — not a horizontal slice of one layer. /to-issues breaks a PRD into these, quizzes you on granularity/dependencies, then publishes in dependency order tagged ready-for-agent.

Prefactor

"Make the change easy, then make the easy change." Restructuring done first (as its own slice) so the real change becomes simple.

implement

The build-stage orchestrator (user-invoked). Takes a PRD or set of issues, drives the implementation, runs typecheck + tests, then calls /review and commits. Reaches for tdd at pre-agreed seams and diagnosing-bugs when things break. Run /tdd directly only for a quick single-feature loop without a PRD.

tdd

A model-invoked discipline: red-green-refactor, one vertical slice at a time. Reached by /implement, not a stage on its own.

Grilling

A relentless interview the agent runs to close the alignment gap before building. The reusable loop behind /grill-me and /grill-with-docs. Rules: tree-structured, one question at a time, always recommend an answer, explore code before asking.

grill-with-docs

The align-stage user-invoked orchestrator. Body is one line: run /grilling using /domain-modeling. Closes failure modes #1 (alignment) and #2 (shared language) in one session.

domain-modeling

The active model-invoked discipline that changes the model: challenges terms, sharpens fuzzy language, stress-tests with scenarios, cross-references code, and writes CONTEXT.md + ADRs inline. Distinct from merely reading the glossary.

Lazy creation

Domain docs are created the moment a term or decision crystallises — never upfront. No CONTEXT.md until the first term resolves; no docs/adr/ until the first ADR is warranted.

The ADR three-gate

Offer an ADR only when all three hold: hard to reverse, surprising without context, the result of a real trade-off. Miss one → skip it.

ask-matt

A user-invoked router over the other user-invoked skills — no process, no code, just a map of how they connect. Invoke when you forget which skill fits. Lists only navigable (user-invoked) skills; model-invoked reflexes are absent by design.

Flow

A path through the skills. The system has one main flow, two on-ramps that merge onto it, and standalone skills off it entirely.

Main flow (idea → ship)

The route most work travels: /grill-with-docs → (optional prototype detour) → branch on multi-session → /to-prd → /to-issues → /implement ×N, or /implement in place.

On-ramp

A starting situation that generates work, then merges onto the main flow. Two of them: /triage (raw bugs/requests you didn't create) and /improve-codebase-architecture (surfaces a deepening idea).

Smart zone

The context window (~120k tokens on SOTA models) within which the model still reasons sharply. Keep grill→prd→issues inside one smart zone; /handoff before you exceed it rather than push on a degraded model.

handoff vs compact

/handoff writes the thread to a markdown file so you continue in a new session (it forks). /compact (built-in) summarises earlier turns in place, same conversation (it continues). Use handoff to branch off (e.g. into a prototype); compact at clean phase breaks.

prototype

A user-invoked detour off the main flow: throwaway code that answers one question. Two branches — a terminal app (logic/state) or several toggleable UI variants. Throwaway from day one; keep only the answer.

— Authoring (writing-great-skills) —

From here down: the vocabulary for writing skills, not just using them. Source: the writing-great-skills SKILL.md + GLOSSARY.md.

Predictability

The root virtue of a skill: the agent takes the same process every run — not the same output. A brainstorm skill should predictably diverge. Every other authoring lever serves this. Avoid: consistency, reliability, robustness, output-determinism.

Context load

The cost a model-invoked skill imposes on the model: its description sits in the window every turn, spending tokens and attention. The brake on splitting into more model-invoked skills. Avoid: token cost, context bloat.

Cognitive load

The cost a user-invoked skill imposes on the human: remembering it exists and when to reach for it (you are the index). Not a cost to zero out — it's the price of human agency. The brake on splitting into more user-invoked skills. Avoid: burden, overhead.

Router skill

A user-invoked skill that names your other user-invoked skills and when to reach for each — one name to remember instead of many. The cure for piled-up cognitive load. It can only hint, never fire them (they have no description). E.g. ask-matt. Avoid: dispatcher, menu, registry.
