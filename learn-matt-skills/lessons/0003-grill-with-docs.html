Lesson 0003 · Alignment + Shared Language

grill-with-docs: the flagship

The first stage of the daily flow, and Matt's "single coolest technique." It closes the alignment gap (failure mode #1) and builds your project's shared language (failure mode #2) — in the same conversation.

Where you are Setup is done — you ran it, and docs/agents/{domain,issue-tracker,triage-labels}.md now exist. The loop can begin. Its first stage is align.

Read the skill — it's three lines

This is the most important thing to see first. The entire grill-with-docs/SKILL.md body is:

Run a /grilling session, using the /domain-modeling skill.

That's it. grill-with-docs is a pure orchestrator — a user-invoked skill that composes two model-invoked disciplines. It's the composition rule made visible: one conductor, two players.

/grill-with-docs (user-invoked conductor — 3 lines) ├── /grilling (model-invoked — the relentless interview loop) └── /domain-modeling (model-invoked — writes CONTEXT.md + ADRs as terms resolve)

So learning grill-with-docs means learning its two players. Here they are.

Player 1 — grilling: the interview loop

The whole skill is one tight instruction. Internalise its four rules:

Interview me relentlessly about every aspect of this plan until we reach a shared understanding. Walk down each branch of the design tree, resolving dependencies between decisions one-by-one. For each question, provide your recommended answer.— grilling/SKILL.md
  • Relentless + tree-structured. It doesn't ask scattershot — it walks the decision tree, resolving dependencies in order.
  • One question at a time. "Asking multiple questions at once is bewildering." (Same pacing rule setup used.)
  • Always recommend an answer. Never a bare "what do you want?" — it proposes, you react. Far faster.
  • Explore code before asking. "If a question can be answered by exploring the codebase, explore the codebase instead." It won't waste your time asking what it can find out itself.

This is the fix for failure mode #1 — "the agent didn't do what I want." The gap closes before a line is written.

Player 2 — domain-modeling: capture the language

Grilling alone produces alignment that evaporates when the session ends. domain-modeling makes it durable by writing two artefacts as the interview happens.

Active, not passive "Merely reading CONTEXT.md for vocabulary is not this skill — that's a one-line habit any skill can do. This skill is for when you're changing the model." Setup's domain.md config is the reading habit; this is the writing discipline.

The five moves it makes during a session

Artefact A — CONTEXT.md (the glossary)

An opinionated glossary. When several words mean one thing, it picks one and lists the rest under _Avoid_:

# Ordering

## Language

**Order**:
A customer's request to purchase one or more items.
_Avoid_: Purchase, transaction

**Customer**:
A person or organization that places orders.
_Avoid_: Client, buyer, account

Three rules keep it sharp: tight (1–2 sentences, define what it IS not what it does), project-specific only (no general programming concepts like "timeout"), and devoid of implementation detail — it's a glossary, never a spec or scratchpad. This is the fix for failure mode #2: concision, every session.

Artefact B — ADRs (the decisions)

An Architectural Decision Record can be one paragraph: "what's the context, what did we decide, why."

# Event-sourced orders

We store orders as an append-only event log rather than mutable rows,
because the audit trail is a hard business requirement and replaying
events lets us rebuild read models cheaply.

Crucially, ADRs are offered sparingly — only when all three are true:

  1. Hard to reverse — changing your mind later costs real effort.
  2. Surprising without context — a future reader will wonder "why on earth this way?"
  3. The result of a real trade-off — genuine alternatives existed; you picked one for reasons.

Miss any one → skip it. Easy-to-reverse? You'll just reverse it. Not surprising? Nobody wonders. No alternative? Nothing to record. This gate is what stops your docs/adr/ from filling with noise.

The discipline tying it together: lazy creation

You never sit down to "write the docs." Both artefacts are created the moment a term or decision crystallises — no CONTEXT.md? It's created when the first term resolves. No docs/adr/? Created when the first ADR is warranted. (This is exactly why setup's domain.md says "proceed silently if the files don't exist" — they're meant to grow lazily.)

When to reach for it

At the align stage — before /to-prd — whenever you're about to build something non-trivial and want to (a) be sure the agent understands you and (b) leave behind language the next session inherits. For a quick non-code plan with no docs to build, plain /grill-me is the lighter tool.
