Lesson 0007 · The Detour Off Step 1

prototype: throwaway code that answers a question

The map (Lesson 6) named this but didn't open it. When grilling hits a question you can't settle in conversation — one that needs state, business logic, or a UI you have to see — you detour into /prototype, learn the answer, and throw the code away.

The one-line definition A prototype is throwaway code that answers a question. The question decides the shape. Get the shape wrong and the whole prototype is wasted — so the first move is never "write code," it's "which question am I answering?"

Step zero: pick the branch

prototype is user-invoked (a deliberate detour, not a reflex). It immediately forks on the kind of question:

LOGIC branchUI branch
Question"Does this logic / state model feel right?""What should this look like?"
Artifacta tiny interactive terminal appseveral UI variants on one route
You dopress keys, watch state changeflip between layouts in the browser
Fitsa backend module / state machinea page or component

If it's genuinely ambiguous and the user is AFK: match the surrounding code — backend → logic, page → UI — and state the assumption at the top of the prototype.

Six rules that bind both branches

  1. Throwaway from day one, marked as such. Live next to the real code (context is obvious) but named so a reader sees it's a prototype.
  2. One command to run. Use the project's existing task runner — the user starts it without thinking.
  3. No persistence by default. State lives in memory. Persistence is what you're checking, not depending on.
  4. Skip the polish. No tests, no error handling beyond runnable, no abstractions.
  5. Surface the state. After every action (logic) or every switch (UI), show the full relevant state.
  6. Delete or absorb when done. Don't leave it rotting in the repo.

LOGIC branch — the portable module + thin shell

The keeper insight, and where this connects to Lesson 5: the logic goes in a pure, portable module; the TUI is a throwaway shell over it.

┌─────────────────────────────────────────┐ │ TUI shell (throwaway) │ │ clear screen → render frame → read key │ │ ▲ │ calls into │ │ │ state out ▼ (one direction) │ │ ┌───────────────────────────────────┐ │ │ │ pure logic module (the KEEPER) │ │ │ │ reducer / state machine / │ │ │ │ pure fns / class — no I/O, no log │ │ │ └───────────────────────────────────┘ │ └─────────────────────────────────────────┘ when answered: delete shell, lift module into real code

Pick the module shape from the question, not from what's easy to wire up: a reducer (state, action) => state for discrete events; a state machine when "which actions are even legal now" is the question; a set of pure functions when there's no current state; a class when it genuinely owns ongoing state. Keep it pure — nothing flows from TUI back into the logic. That purity is exactly what lets the validated module be lifted into the real codebase later. (This is a deep module being prototyped: small interface, hidden behaviour — see Lesson 5.)

The TUI: on every tick console.clear() and re-render the whole frame — current state (one field per line) then the keyboard shortcuts [a] add [t] tick [q] quit. One stable screen, never growing scrollback.

UI branch — variants behind ?variant=

Generate several radically different variants on one route, switchable from a floating bar. Strongly prefer:

  • Sub-shape A (default): render variants inside an existing page, gated by a ?variant= URL param — real header, real data, real density. A variant judged in a vacuum always looks fine.
  • Sub-shape B (last resort): a throwaway route, only when the thing has no existing page to live in. Follow the project's routing convention; put "prototype" in the path.

Default to 3 variants, cap at 5. They must be structurally different — different layout, hierarchy, primary affordance — not recoloured. The floating switcher: ←/label/→, updates the URL (shareable, reload-stable), arrow-key support, and hidden in production builds so a stray merge can't ship it.

The feedback you're hunting for The valuable response is rarely "B wins." It's "I want the header from B with the sidebar from C" — that recombination is the actual design the user wants. Same spirit in LOGIC: "wait, that shouldn't be possible" is a bug in the idea, which is the whole point.

When done: keep the answer, kill the code

The answer is the only thing worth keeping from a prototype — never the code. Capture it somewhere durable (commit, ADR, issue, or a NOTES.md beside the prototype) with the question it answered. Then delete the throwaway, or fold the validated decision into real code. If you're AFK, leave the placeholder so the verdict gets filled in before deletion.

And recall the bridge: you reached this detour via /handoff out of the grill thread, and you carry the answer back with /handoff in — referenced from the original idea (Lesson 6).

Check yourself

What decides a prototype's shape?

"The question decides the shape." Logic question → terminal app; look-and-feel question → UI variants. Picking by what's easy to build wastes the prototype.

In the LOGIC branch, which part is worth keeping?

The TUI shell is throwaway; the pure reducer/state-machine/function module is portable and gets lifted into the real codebase. Keep them strictly separate.

The single thing worth keeping from any prototype is…

Throwaway code, durable answer. Capture the answer (and the question) in a commit/ADR/NOTES.md, then delete or absorb the code.
