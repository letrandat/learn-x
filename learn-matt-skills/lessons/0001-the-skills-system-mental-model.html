Lesson 0001 · Mental Model

Matt Pocock's skills, in one mental model

Before any single skill makes sense, you need the shape of the whole thing: small composable prompts, one invocation axis, four failure modes they fight.

What a "skill" actually is

A skill is not code and not a framework. It is a folder containing a SKILL.md — a markdown file with YAML frontmatter and a body of instructions — plus optional bundled files. When you type /name (or the agent reaches for it), that body is loaded into the agent's context and steers how it behaves for that task.

Matt contrasts this with GSD, BMAD, and Spec-Kit:

These skills are designed to be small, easy to adapt, and composable. They work with any model. Hack around with them. Make them your own. — mattpocock/skills, README

Frameworks own your process and hide their bugs. Skills hand you a stack of short, readable prompts you can edit. Install them with:

npx skills@latest add mattpocock/skills

The one axis that organises everything

Every skill splits on a single question: who is allowed to invoke it?

KindInvoked byJobExamples
user-invokedonly you, typing /nameorchestrate a flow/grill-me, /triage, /to-prd
model-invokedyou or the agent, automaticallyhold reusable disciplinetdd, diagnosing-bugs, domain-modeling

The mechanism is one frontmatter line. disable-model-invocation: true makes a skill user-only. The composition rule:

A user-invoked skill may invoke model-invoked skills, but never another user-invoked one.— README, Reference section

This keeps orchestration flat: one conductor (the slash command you typed) calling reusable players (the disciplines). No nested conductors fighting for control of the conversation.

The four failure modes they fight

Matt frames the whole repo as a response to four ways agents disappoint you. This is the "why" behind every skill — learn it and the catalogue organises itself.

#Failure modeRoot causeThe fix (skill)
1"Didn't do what I want"misalignment — you didn't fully say it; the agent didn't fully askgrill-me / grill-with-docs
2"Way too verbose"no shared language — agent reinvents jargon every sessionCONTEXT.md via grill-with-docs + domain-modeling
3"The code doesn't work"no feedback loop — agent flies blindtdd, diagnosing-bugs
4"We built a ball of mud"entropy — agents accelerate complexityimprove-codebase-architecture + codebase-design
The big idea behind #2 A shared glossary lets the agent say "a problem with the materialization cascade" instead of "a problem when a lesson inside a section of a course is made real." Same meaning, a fraction of the tokens — every session. Matt calls this possibly the single coolest technique in the repo.

How it fits: the daily flow

The skills chain into one loop, from idea to merged code. Each stage feeds the next:

align ─────────► plan ──────► slice ───────► build ──────────► maintain /grill-with-docs /to-prd /to-issues /implement /improve-codebase-architecture builds CONTEXT.md vertical │ + ADRs while you slices → ├─ uses tdd (model-invoked) get interviewed grabbable ├─ uses diagnosing-bugs (model-invoked) issues └─ calls /review, then commits

Notice the build stage is itself the axis in miniature: /implement is the conductor — it takes the PRD/issues, drives the work, then runs /review and commits. It reaches for tdd at pre-agreed seams and diagnosing-bugs when something breaks. One conductor, reusable players — the composition rule at work. (Run /tdd directly only for a quick one-feature loop with no PRD.)

One stage is missing from that picture because it runs before the loop ever starts: setup-matt-pocock-skills. It writes the per-repo config the other skills assume. That's Lesson 2.

Check yourself

Why can't a user-invoked skill call another user-invoked skill?

Flat orchestration: the slash command you type is the single conductor; it may call reusable model-invoked disciplines, never a second conductor.

What single line makes a skill user-invoked only?

disable-model-invocation: true tells the agent never to auto-reach for the skill — you must type its slash command.
