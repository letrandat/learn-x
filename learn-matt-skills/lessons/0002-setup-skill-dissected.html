Lesson 0002 · The Bootstrap Skill

setup-matt-pocock-skills, dissected

The one skill you run first, taken apart: its file anatomy, the three configs it writes, and the design decisions that make it a model for every other skill.

The job in one sentence Answer three per-repo questions once, write the answers to docs/agents/*.md, and point your CLAUDE.md at them — so every other engineering skill reads consistent config instead of re-guessing each session.

Component 1 — file anatomy

The skill is a folder of six files. Five are seed templates; one is the brain.

setup-matt-pocock-skills/ ├── SKILL.md ← the brain: frontmatter + 5-step process ├── issue-tracker-github.md ← seed template (GitHub flavour) ├── issue-tracker-gitlab.md ← seed template (GitLab flavour) ├── issue-tracker-local.md ← seed template (local-markdown flavour) ├── triage-labels.md ← seed template (label mapping table) └── domain.md ← seed template (domain-doc consumer rules)

The frontmatter is a contract

---
name: setup-matt-pocock-skills
description: Configure this repo for the engineering skills — set up its
             issue tracker, triage label vocabulary, and domain doc layout.
             Run once before first use of the other engineering skills.
disable-model-invocation: true
---
  • name becomes the slash command /setup-matt-pocock-skills.
  • description is the only part loaded into context until the skill fires — so it states what it does + when ("Run once before first use"). This is progressive disclosure: the agent reads cheap descriptions of all skills, pulls a full body only on a match.
  • disable-model-invocation: true makes it user-invoked only. Setup is a one-time, file-writing ceremony — you never want it firing silently mid-task.

The body is a conversation script, not a script

This is a prompt-driven skill, not a deterministic script. Explore, present what you found, confirm with the user, then write.— SKILL.md, opening line

That sentence is the whole philosophy in miniature. The body is a 5-step process the agent runs with judgement: Explore → Present & ask → Confirm → Write → Done. The five other .md files are bundled resources — referenced by relative link, read only at the Write step, then copied into your repo as starting content.

Component 2 — the three per-repo configs

Everything setup does serves three outputs under docs/agents/. Learn what each answers and who reads it.

A · Issue tracker → docs/agents/issue-tracker.md

Answers: where do issues live, and how do I touch them?
Read by: /triage, /to-issues, /to-prd — they need "publish to the issue tracker" to mean a concrete command.

Setup proposes a default by sniffing git remote -v, then offers:

  • GitHub — the gh CLI
  • GitLab — the glab CLI
  • Local markdown — files under .scratch/<feature>/ (solo / no remote)
  • Other (Jira, Linear…) — you describe it in a paragraph, recorded as prose

If (and only if) GitHub or GitLab: one follow-up — "PRs as a request surface? (default no)". If yes, /triage also pulls external PRs into the queue. The seed template stores it as a literal switch:

PRs as a request surface: no. (Set to yes if this repo treats external PRs as feature requests; /triage reads this flag.)— issue-tracker-github.md

The template is self-documenting config: the setting and the explanation of what reads it live in the same file. It even bakes in GitHub's gotcha — issues and PRs share a number space, so #42 is ambiguous; resolve with gh pr view 42, fall back to gh issue view 42.

B · Triage labels → docs/agents/triage-labels.md

Answers: what label strings does this repo actually use?
Read by: /triage, which runs issues through a five-state machine. Without a mapping it would create duplicate labels.

┌──────────────┐ new ──────► │ needs-triage │ maintainer must evaluate └──────┬───────┘ ┌─────────────┼──────────────┬───────────────┐ ▼ ▼ ▼ ▼ needs-info ready-for-agent ready-for-human wontfix (ask (fully spec'd, (needs a (dead) reporter) AFK-ready) human)

The seed file is a two-column table: canonical role on the left, your real label string on the right (defaults equal). The standout state is ready-for-agent — "so completely specified an AFK agent can grab it with zero human context." That's the endgame of the whole pipeline.

C · Domain docs → docs/agents/domain.md

Answers: where's the shared language, and how should skills consume it?
Read by: improve-codebase-architecture, tdd, diagnosing-bugs — they read CONTEXT.md + docs/adr/ before exploring code.

The only choice is layout: single-context (one CONTEXT.md at root, most repos) vs multi-context (CONTEXT-MAP.md pointing at per-context files, monorepos). Three rules the seed file bakes in:

  1. Proceed silently if the files don't exist. Don't nag — /domain-modeling creates them lazily when a term is actually resolved.
  2. Use the glossary's vocabulary in every output. A concept not in the glossary is a signal: you're inventing language (stop) or there's a real gap (flag it).
  3. Flag ADR conflicts explicitly — "Contradicts ADR-0007 — but worth reopening because…" — never silently override.

Component 3 — the design details worth stealing

Read setup as a worked example of how to write any good skill. Five seams:

PrincipleHow setup does it
Config-as-markdown, repo-ownedOutput lives in your committed docs/agents/. Skills stay stateless; the repo holds state. Hand-edit later without re-running.
Teach, don't assume"Assume the user does not know what these terms mean." Each section opens with an explainer.
One decision at a time"Walk the user through the three decisions one at a time… Don't dump all three at once."
Defaults from evidenceReads git remote -v to propose GitHub vs GitLab rather than asking cold.
Idempotent, non-destructive writesEdit CLAUDE.md if present, else AGENTS.md, else ask. Update an existing ## Agent skills block in place — never append a duplicate; never clobber surrounding edits.
Confirm before writeShow a draft of the block + all three docs, let the user edit, then write. Reversible by design.

What you end up with

CLAUDE.md └── ## Agent skills ← three one-line pointers docs/agents/ ├── issue-tracker.md ← config A ├── triage-labels.md ← config B └── domain.md ← config C

From here, the daily flow from Lesson 1 has the config it needs to run.

Check yourself

Why does setup write config to files instead of just remembering it?

Skills are stateless prompts. Persisting config to docs/agents/ lets every future skill run — and every other skill — read the same answers, and lets you hand-edit later.

Which file does setup edit when both CLAUDE.md and AGENTS.md exist?

Precedence: CLAUDE.md wins if it exists. It never creates AGENTS.md when CLAUDE.md is already there, and never clobbers the other.

What does the ready-for-agent triage label promise?

ready-for-agent means the issue is so fully specified an autonomous (away-from-keyboard) agent can pick it up with zero extra human context.
