Lesson 0004 · The Assembly Line

plan → slice → build: to-prd · to-issues · implement

Alignment is done. Now the three skills that turn a shared understanding into merged code — an assembly line where each stage hands the next a cleaner artefact, and your setup config threads through all of them.

The thread that ties them together All three skills open with the same line: "The issue tracker and triage label vocabulary should have been provided to you — run /setup-matt-pocock-skills if not." This is why Lesson 2 came first. Every stage reads your docs/agents/*.md — the tracker to publish to, the labels to apply, the glossary to speak in.

Stage 1 — to-prd: synthesise, don't interview

The single most important thing about to-prd is what it refuses to do:

Do NOT interview the user — just synthesize what you already know.— to-prd/SKILL.md

This is the deliberate complement to Lesson 3. grill-with-docs does all the asking; to-prd does none. By the time you run it, the conversation already holds the decisions — its job is to write them down, not re-extract them. Grill to align; /to-prd to record.

Its three steps

  1. Explore the codebase, using the domain glossary vocabulary and respecting ADRs in the area you touch. (Lesson 3's artefacts paying off.)
  2. Sketch the test seams — and check them with you. The rule: "Prefer existing seams. Use the highest seam possible. The fewer seams, the better — the ideal number is one." A seam is where you'll test the feature; fewer/higher seams = simpler, more durable tests.
  3. Write the PRD and publish it to your tracker, tagged ready-for-agent — no extra triage needed.

What's in the PRD (and what's banned)

SectionHolds
Problem Statementthe problem, from the user's POV
Solutionthe solution, from the user's POV
User Storiesa long numbered list — "As an X, I want Y, so that Z"
Implementation Decisionsmodules, interfaces, schema, API contracts, arch decisions
Testing Decisionswhat makes a good test, which modules, prior art
Out of Scope / Notesboundaries + anything else
The banned thing "Do NOT include specific file paths or code snippets. They may end up being outdated very quickly." A PRD records decisions, not code — decisions age slowly, code ages fast. (One exception: a prototype snippet that encodes a decision more precisely than prose — a state machine, schema, type shape — trimmed to just the decision.)

Stage 2 — to-issues: vertical slices

A PRD is one big lump. to-issues cuts it into independently-grabbable issues — and the shape of the cut is the whole lesson.

Tracer bullets, not layers

Each issue is a thin vertical slice that cuts through ALL integration layers end-to-end, NOT a horizontal slice of one layer.— to-issues/SKILL.md
HORIZONTAL (wrong) VERTICAL / tracer bullet (right) ┌──────────────────┐ ┌──┐ ┌──┐ ┌──┐ │ all the UI │ │UI│ │UI│ │UI│ ├──────────────────┤ ├──┤ ├──┤ ├──┤ │ all the API │ │API│ │API│ │API│ each slice = one ├──────────────────┤ ├──┤ ├──┤ ├──┤ complete path, │ all the schema │ │DB│ │DB│ │DB│ demoable on its own └──────────────────┘ └──┘ └──┘ └──┘ one giant unverifiable step slice1 slice2 slice3

Each slice delivers a narrow but complete path through every layer (schema → API → UI → tests) and is demoable on its own. That's a "tracer bullet": fire one thin shot all the way to the target, see it land, then widen. It connects straight to failure mode #3 — you get working, verifiable feedback after every slice instead of at the end.

Two more moves

  • Prefactor first. "Make the change easy, then make the easy change." Any restructuring that makes the slices simpler is its own first slice.
  • Quiz, then publish. It presents the breakdown as a numbered list (title · blocked-by · user-stories-covered) and asks: granularity right? dependencies right? merge/split anything? It iterates until you approve — then publishes in dependency order (blockers first, so real IDs can be referenced), each tagged ready-for-agent.

Stage 3 — implement: build it

You met this one in Lesson 1's correction. It's the user-invoked build conductor, and its whole body is five instructions:

Implement the work described in the PRD or issues.
Use /tdd where possible, at pre-agreed seams.
Run typechecking + single test files regularly, full suite once at the end.
Once done, use /review.
Commit to the current branch.

Note how it consumes the previous stages' output: the PRD/issues tell it what to build, the pre-agreed seams (from to-prd step 2) tell tdd where to test, and the ready-for-agent label is the handoff token that said these were ready. One conductor (/implement) reaching for one player (tdd), then /review, then commit.

The whole line, in one view

grill-with-docs ──► to-prd ──────► to-issues ──────► implement ──► /review ──► commit (Lesson 3) synthesise vertical drives aligns + decisions slices build, writes CONTEXT into a PRD (tracer uses tdd + ADRs + label bullets) at seams │ │ │ │ └─ glossary ───────┴─ labels ──────┴─ tracker ──────┘ all from docs/agents/*

Each arrow narrows ambiguity: a conversation → a recorded spec → grabbable slices → working code. The ready-for-agent label rides along the whole way — it's what makes the slices safe to hand to an AFK agent.

Check yourself

Why does to-prd deliberately refuse to interview you?

to-prd is the complement of grill-with-docs: grilling extracts the decisions, to-prd just synthesises and records what the conversation already holds.

What is a tracer-bullet (vertical) slice?

A thin slice cutting end-to-end through schema, API, UI and tests — demoable on its own — not a horizontal slice of a single layer.

Why are file paths banned from a PRD?

A PRD records durable decisions. Specific paths and snippets go stale quickly, so they're excluded (save the rare prototype snippet that encodes a decision).
