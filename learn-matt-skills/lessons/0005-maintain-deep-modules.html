Lesson 0005 · Maintain — Fighting the Ball of Mud

maintain: deep modules & the architecture review

The stage you run every few days, not once per feature. It fights failure mode #4 — entropy — with a shared design vocabulary (codebase-design) and a visual review that grills you toward deeper modules (improve-codebase-architecture).

Closing the loop You now have the whole daily flow: setup → align → plan → slice → build. But shipped code rots. Agents accelerate entropy — they add complexity fast. This stage is the counter-pressure: invest in design continuously, not as a rescue mission.

Foundation — codebase-design: the shared vocabulary

Before you can improve architecture you need words for it. codebase-design is a model-invoked vocabulary skill — it holds no process, just language, used wherever code is designed. The rule is strict: use these terms exactly — don't substitute "component," "service," "API," or "boundary."

TermMeansAvoid
Moduleanything with an interface + implementation (fn, class, package, slice) — scale-agnosticunit, component, service
Interfaceeverything a caller must know: types, invariants, ordering, error modes, perf — not just the signatureAPI, signature
Depthleverage at the interface: behaviour exercised per unit of interface learned—
Seam (Feathers)a place you can alter behaviour without editing there; where the interface livesboundary
Adaptera concrete thing satisfying an interface at a seam (role, not substance)—
Leveragewhat callers gain from depth: more capability per unit learned—
Localitywhat maintainers gain: change/bugs/knowledge concentrate in one place—

Deep vs shallow — the whole idea in one picture

DEEP (the goal) SHALLOW (avoid) ┌───────────────────┐ ┌─────────────────────────────────┐ │ small interface │ │ large interface │ ├───────────────────┤ ├─────────────────────────────────┤ │ │ │ thin implementation │ │ lots of hidden │ └─────────────────────────────────┘ │ implementation │ interface ≈ implementation → │ │ it barely earns its keep └───────────────────┘

A deep module hides a lot of behaviour behind a small interface. A shallow one has an interface nearly as complex as its implementation — it just passes work through. Design asks three questions: fewer methods? simpler params? hide more complexity inside?

Four principles worth memorising

  1. The deletion test. Imagine deleting the module. If complexity vanishes, it was a pass-through (shallow). If complexity reappears across N callers, it was earning its keep. "Concentrates complexity" is the signal you want.
  2. Depth is a property of the interface, not the implementation. A deep module can be internally made of small swappable parts — they just aren't in the interface.
  3. The interface is the test surface. Callers and tests cross the same seam. If you need to test past the interface, the module is the wrong shape. (This is the seam your to-prd chose and tdd tested against — same concept.)
  4. One adapter = a hypothetical seam; two = a real one. Don't introduce a seam unless something actually varies across it.
Why "depth-as-leverage", not Ousterhout's ratio Matt explicitly rejects depth-as-(impl-lines ÷ interface-lines) — it rewards padding the implementation. Depth here means leverage: capability per unit of interface learned. (Also rejected: "interface" = the TS keyword; "boundary" = DDD's bounded context.)

The orchestrator — improve-codebase-architecture

This user-invoked skill hunts for deepening opportunities — refactors that turn shallow modules into deep ones, for testability and AI-navigability. It's the busiest composition in the repo:

/improve-codebase-architecture (user-invoked conductor) ├── /codebase-design ← the architecture vocabulary (above) ├── Explore subagent ← walks the codebase noting friction ├── /grilling ← designs the candidate you pick └── /domain-modeling ← captures terms/ADRs as decisions crystallise reads: CONTEXT.md (domain names) + docs/adr/ (decisions not to re-litigate)

Its three steps

  1. Explore. Read CONTEXT.md + ADRs first, then send the Explore subagent through the code noting friction — bouncing between many small modules, shallow modules, pure functions extracted for testability while the real bugs hide in how they're called (no locality), leaky seams, untestable interfaces. Apply the deletion test to anything suspicious.
  2. Present a visual HTML report. Written to the OS temp dir, never the repo ($TMPDIR → fallback /tmp), then opened for you. Tailwind + Mermaid. Each candidate is a card: Files · Problem · Solution · Benefits (in locality/leverage terms) · before/after diagram · a strength badge (Strong / Worth exploring / Speculative). Ends with a Top recommendation. It does not propose interfaces yet — it asks which one you want to explore.
  3. Grilling loop. Once you pick, it runs /grilling (Lesson 3!) to walk the design tree — what sits behind the seam, what tests survive — with /domain-modeling side effects inline: new module name not in CONTEXT.md? add it. Reject a candidate for a load-bearing reason? offer an ADR so future reviews don't re-suggest it.
You've seen this report format before "Write a self-contained, visual HTML file to temp and open it." That's exactly how these lessons are built — the same visual-artefact pattern Matt favours for anything a human needs to look at and compare.

How it closes the loop

align → plan → slice → build ──────────► ship ▲ │ │ ▼ └──────── maintain (every few days) ◄────┘ /improve-codebase-architecture deepen modules · keep CONTEXT.md + ADRs current

The maintain stage feeds back into align: its grilling sharpens CONTEXT.md and records ADRs, so the next feature starts from a cleaner model and a deeper codebase. That feedback is the whole point — design invested every day, not entropy paid down in a big-bang rewrite.

Check yourself

What makes a module "deep"?

Depth = leverage: a large amount of behaviour exercised through a small interface. Matt rejects the lines-ratio definition because it rewards padding the implementation.

A module passes the deletion test when removing it…

If deleting it makes complexity reappear across N callers, it was earning its keep (deep). If complexity just vanishes, it was a shallow pass-through.

Where does the architecture report get written?

To $TMPDIR (fallback /tmp), then opened for you — deliberately nothing lands in the repo. Each run gets a fresh timestamped file.
