Predictability & the two loads

You've spent seven lessons using Matt's skills. Now you slide into the author's seat. The whole craft rests on one virtue and one tradeoff — get these, and every later decision (what to disclose, when to split, which words to pick) is just this tension applied again.

The thesis — the one sentence A skill exists to wrangle determinism out of a stochastic system. Its root virtue is predictability: the agent takes the same process every run — not the same output. A brainstorm skill should predictably diverge; its tokens vary, its behaviour doesn't. Every lever below serves predictability.

The flip: from picking the axis to owning it

In Lesson 1 you learned to read the user-vs-model axis. As an author you now choose it — and the choice is forced, because the description field is the axis. Keep it and the skill is model-invoked; delete it and the skill is user-invoked. There is no third state.

The reason the choice matters is that each option spends a different currency. Authoring is mostly the art of deciding which of two loads to pay.

The two loads

Context loadCognitive load
Who paysthe modelyou, the human
What it isthe description sitting in the window every turn — tokens and attentionremembering the skill exists and when to reach for it (you are the index)
Paid bymodel-invoked skillsuser-invoked skills
Brake onsplitting into more model-invoked skillssplitting into more user-invoked skills

Cognitive load is not a cost to minimise to zero — it's the price of human agency, the reason some skills stay user-invoked. Spend it where your judgement matters; remove it where it doesn't.

The invocation decision

Does the AGENT need to reach this skill on its own (or must ANOTHER skill be able to reach it)? │ yes │ no ▼ ▼ ┌──────────────────┐ ┌──────────────────┐ │ MODEL-INVOKED │ │ USER-INVOKED │ │ keep description│ │ strip it │ │ agent + skills │ │ only you, by │ │ can fire it │ │ name, can fire │ │ pays context │ │ zero context, │ │ load FOREVER │ │ spends YOUR head│ └──────────────────┘ └──────────────────┘

The rule, stated plainly: pick model-invocation only when the agent must reach the skill itself, or another skill must. If it only ever fires by hand, make it user-invoked and pay no context load. Note the asymmetry — a description only ever adds agent discovery; the human can always type the name. So model-invocation is user-reach plus agent-reach, never agent-only.

Why other skills can only reach model-invoked ones Reachability rides on the description. A model-invoked skill has one, so another skill can fire it — which is why shared reference often lives in a model-invoked skill. A user-invoked skill has no description, so nothing but you can reach it — not even another skill.

When cognitive load piles up: the router skill

Go all-in on user-invoked skills and you eventually can't remember what you have. That piled-up cognitive load has a specific cure: a router skill — one user-invoked skill that names the others and says when to reach for each. You've already met one: ask-matt. A router can only hint, never fire them (no descriptions to fire) — but it gives you one name to remember instead of many.

Check yourself

Predictability in a skill means the agent reliably produces…

Same process, not same output. A brainstorm skill should predictably diverge — its behaviour is fixed even as its tokens vary.

Deleting a skill's description field makes it…

The description IS the axis. Strip it → user-invoked: only you can fire it by name, and it adds nothing to the context window every turn.

Choose model-invocation precisely when…

The trigger is agent (or other-skill) reach, not frequency or size. If it only ever fires by hand, stay user-invoked and pay no context load.

Too many user-invoked skills to remember is cured by a…

Piled-up cognitive load → a router skill (e.g. ask-matt): one user-invoked skill naming the others and when to use each. It hints; it can't fire them.
